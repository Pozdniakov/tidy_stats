Источник правды для всех стилевых решений книги. При любых сомнениях сверяться здесь.

## Часть I. Текст и стиль

### 1. Голос и стиль

**Местоимения.** Книга использует оба — оба корректны:
- **«мы»** — автор и читатель делают что-то вместе: «давайте посмотрим», «мы получим»
- **«вы»** — обращение к читателю как к деятелю: «вы можете попробовать»

«вы» — всегда строчными: **вы**, **вам**, **вас**. Не «Вы», «Вам», «Вас».

**Консервативная редактура.** Если фраза понятна и не нарушает конвенций — не трогать. Редактировать только то, что реально улучшает читаемость, уменьшает неоднозначность или исправляет ошибку. Не переписывать ради «более красивого» варианта.

**Авторский стиль.** Книга написана дружелюбно и разговорно. Шутки, живые комментарии, разговорные обороты — сохранять, если они не мешают пониманию. Не делать текст чрезмерно академичным.

**Залог.** Предпочтительно активный: «функция возвращает» вместо «функцией возвращается». Пассив допустим, когда подлежащее неважно, но «является» и «используется» — слова-паразиты, заменять по возможности.

**Время при описании кода.** При документировании поведения — настоящее: «`is.na()` возвращает `TRUE` для `NA`». При совместном рассуждении с читателем — будущее нормально: «запустим — что получим? Функция вернёт `TRUE`».

**«Как видно из рисунка»** — избегать: при изменении порядка рисунков фраза теряет смысл. Использовать cross-reference: «(@fig-label)».

**«Заметьте» / «Обратите внимание»** — допустимы в прозе для небольших inline-оговорок. Если замечание важное и читатель не должен его пропустить — выносить в `callout-important` (*Осторожно:*). Ориентир: если «Заметьте» можно убрать без потери смысла — убрать или заменить запятой.

### 3. Тире

**Em dash (длинное тире):** ` --- ` с пробелами — конвенция Quarto/Pandoc. Не использовать ` - ` или ` -- ` в прозе.

**En dash (короткое тире):** символ `–` напрямую — для числовых диапазонов: «стр. 12–18», «главы 305–370», «1990–2020». Не путать с em dash и дефисом.

**Исключение:** ` - ` внутри путей по меню в бэктиках (`` `Tool - Global Options...` ``, см. §13) — это разделитель уровней меню, а не тире. Не конвертировать. При sed-проходах по ` - ` исключать строки с путями по меню.

**Двойные пробелы** — не использовать. Найти: `grep -Pn ' {2,}' file.qmd`.

⚠️ **Самый массовый дефект книги:** ` -- ` встречается значительно чаще, чем ` --- `. При редактировании любой главы:
```bash
# Linux
sed -i 's/ -- / --- /g' NNN-chapter.qmd
# macOS
sed -i '' 's/ -- / --- /g' NNN-chapter.qmd
```

### 4. Числа в тексте

Числа записываются **цифрами** — книга технического характера. Исключение: устойчивые обороты («два в одном») и числа в начале предложения (переформулировать или написать словом).

Дробные числа — с **точкой**: `0.5`, `3.14`. Запятая как десятичный разделитель не используется.

### 5. Списки

- **Нумерованный список** — только когда порядок важен (шаги установки, этапы алгоритма).
- **Маркированный список** — перечисления без порядка.

Список из 2 элементов — рассмотреть, не лучше ли написать в строку: «X и Y».

### 6. Структура главы

Каждая глава должна иметь:
1. **Вводный абзац** — о чём глава, зачем читателю это нужно.
2. **Основное содержание.**
3. **Итоговый абзац** — что узнали, что дальше.

Глава без итогов — флаг при вычитке.

**Порядок введения понятий:** понятие не должно использоваться до того, как введено. Если термин встречается раньше своего определения — перестановка или forward-reference.

## Часть II. Форматирование технических объектов

### 7. Термины

При **первом появлении** технического термина — жирным с переводом в курсиве:

```
**термин** *(english translation)*
```

Примеры: **дисперсия** *(variance)*, **медиана** *(median)*, **статистика вывода** *(inferential statistics)*.

**Устоявшиеся русские термины** — предпочтительные формы:

| Английский | Правильно | Неправильно |
|---|---|---|
| data frame | датафрейм | дата-фрейм, data frame |
| tibble | тиббл | тибл, tibble |
| pipe operator | пайп (существительное) | пайп-оператор, «пайпить» (глагол-сленг) |
| NA / missing values | пропущенные значения | NA в русском тексте |
| confidence interval | доверительный интервал | — |
| null hypothesis | нулевая гипотеза | — |
| linear regression | линейная регрессия | — |

**Когда добавлять перевод:**
- Нужен — статистические концепции, R-концепции, типы данных R, типы шкал, ML/Data Science: всё, с чем студент столкнётся в английской литературе.
- Не нужен — простые описательные фразы: **вектора длиной 1**.
- Граничный случай: **среднее** — слишком базовое, перевод опускается.

**Повторные упоминания:** жирный не повторять. Жирный во второй раз и далее — ошибка.

**Нежелательный формат** `***Термин *(перевод)****`: вложенный курсив внутри жирного курсива ломается в PDF/EPUB и создаёт визуальный шум. При редактировании заменять на формат выше.

### 8. «Параметр» vs «аргумент»

> Подробное исследование вопроса — в `research-notes.md` (запись от 2026-06-05). Краткий итог ниже.

Терминология здесь расходится между общим программированием и самим R:

- **Общая CS-традиция** (и многие R-туториалы): **параметр** — имя в *определении* функции; **аргумент** — *значение* при вызове. По ней `sep =` — параметр.
- **Собственная терминология R** (R Language Definition, Hadley «Advanced R»): имена в определении — **«формальные аргументы»** *(formal arguments)*, значения при вызове — **«фактические аргументы»** *(actual / supplied arguments)*. Слово «parameter» R для входов функции **не использует** (раздел хелпа называется `Arguments`).

**Решение для книги.** Используем привычное **«параметр»** для именованного входа функции (`параметр sep =`) — это распространённое употребление, понятное новичку. **Не** проводить массовую замену «параметр» → «аргумент» в коде/прозе. Соответствие R-терминологии («параметр» ≈ «формальный аргумент») один раз поясняется в callout «Для продвинутых» там, где впервые подробно вводятся аргументы функций.

- **«аргумент»** — корректно, когда речь о конкретном *переданном значении*: «передайте функцию как аргумент», «вызвать `getwd()` без аргументов».
- **«параметр»** — для именованного слота функции (`na.strings =`, `sep =`) и для статистики (параметры распределения, μ, σ). Коллизия этих двух смыслов не критична — контекст (программирование vs статистика) всегда однозначен.

На будущее: возможно, позднее перейти полностью на «аргумент» (ближе к терминологии R) — пока остаётся «параметр».

### 10. Функции и операторы

| Контекст | Формат | Пример |
|---|---|---|
| Вызов функции | `` `функция()` `` | `mean()`, `read_csv()` |
| Функция как объект (без вызова) | `` `функция` `` | передаём `mean` в `FUN =` |
| Семейство функций | `` `apply()` `` (без `*`) | семейство `apply()` |
| Оператор | `` `оператор` `` | `%>%`, `\|>`, `::` |

**Нотация `package::function()`** — только в двух случаях:
1. Первое упоминание функции из незагруженного пакета: `data.table::fread()`
2. Разрешение конфликта имён: `dplyr::filter()` vs `stats::filter()`

В остальных случаях — просто `функция()`.

**Ключевые слова и конструкции — не операторы.** По `?Reserved` зарезервированные (ключевые) слова R: `if`, `else`, `repeat`, `while`, `function`, `for`, `in`, `next`, `break` (а также `TRUE`/`FALSE`/`NULL`/`NA`/`Inf`/`NaN`). Их **нельзя называть «операторами»**:
- `if`/`else`/`else if` — **условная конструкция**; `for`/`while`/`repeat` — **цикл**; `function` — **ключевое слово**.
- **Оператор** — это `+`, `-`, `*`, `/`, `^`, `%%`, `==`, `!=`, `<`, `&`, `|`, `!`, `<-`, `=`, `->`, `:`, `::`, `$`, `[`, `~`, `%in%`, `|>`, `%>%` и другие инфиксные/специальные операторы (технически — тоже функции).

Подробности и источники — в `research-notes.md` (запись от 2026-06-05).

### 14. Математика и статистика

**Inline-формулы:** `$x$` — переменные, символы, короткие выражения в тексте.  
**Display-формулы:** `$$...$$` — формулы на отдельной строке.  
**Греческие буквы** — только в LaTeX: `$\mu$`, `$\sigma$`, `$\alpha$`. Не вставлять Unicode (µ, σ) напрямую.

**Параметры vs статистики:** $\mu$, $\sigma$ — параметры генеральной совокупности; $\bar{x}$, $s$ — выборочные статистики. Не смешивать.

**Статистические символы в тексте** — курсивом:

| Символ | Формат | Пример |
|---|---|---|
| F-статистика | `*F*` | значение *F* = 3.4 |
| t-статистика | `*t*` | при *t*(48) = 2.1 |
| размер выборки | `*n*` | при *n* = 100 |
| корреляция | `*r*` | коэффициент *r* = 0.6 |

**p-value** — исключение: пишется как `p-value` (не курсивом, не в LaTeX).

**Форматирование значений p** — с нулём перед точкой:
```
p = 0.03   p < 0.001
```
APA-стиль без нуля (`.03`) не используется — он нетипичен для русскоязычного учебника. В R-коде и выводе функций — тот же формат.

**Аббревиатуры** (ANOVA, GLM, OLS, PCA) — без курсива. При первом упоминании расшифровка в скобках или сноске.

## Часть III. R-код

### 15. Стиль R-кода

**Пайп.** В книге используются оба:
- `|>` — нативный (R ≥ 4.1), встречается в новых главах
- `%>%` — из `{magrittr}`, встречается в ранних главах

При написании нового кода — `|>`. При редактировании существующего — сохранять тот пайп, который уже используется в файле. Специальные операторы `{magrittr}` (`%T>%`, `%<>%`, `%$%`) оставлять как есть — нативного эквивалента нет.

**Именование объектов** — snake_case: `my_data`, `mean_age`. camelCase не использовать.

**Кавычки в R-коде** — двойные: `"значение"`. Одинарные — только если строка содержит двойные кавычки: `'Он сказал "привет"'`.

**`if_else()` вместо `ifelse()`** — кроме глав по базовому R (`040-control_flow.qmd`, `910-tasks.qmd`), где `ifelse()` намеренно.

**`set.seed(42)`** — стандартное зерно для всего нового кода со случайностью. Не менять число.

**Длинные выводы** — обрезать через `head()`.

### 16. Комментарии в коде

Комментарии — часть учебного материала:
- С заглавной буквы, без точки в конце.
- На русском языке.
- Описывают *что* делает код, а не *как*.
- Соответствуют коду рядом (не устаревшие, не вводящие в заблуждение).

```r
# Загружаем данные
heroes <- read_csv("data/heroes_information.csv")

# Считаем среднее по группам
heroes |> group_by(Alignment) |> summarise(mean_height = mean(Height, na.rm = TRUE))
```

### 17. Чанки кода

**Предпочтительный синтаксис** — YAML-опции через `#|`:
```r
#| echo: false
#| message: false
```
Старый синтаксис `` ```{r, eval = FALSE} `` — устаревший, мигрировать при редактировании глав.

**Метки чанков (`label`)** — по умолчанию не использовать ради однородности. Обычные чанки кода оставлять без метки: `` ```{r} ``, а не `` ```{r my-chunk} `` или `#| label: my-chunk`. Метка нужна только тогда, когда на чанк ссылаются как на рисунок или таблицу — тогда `#| label: fig-...` / `#| label: tbl-...` (см. ниже). При редактировании глав удалять метки у нессылаемых чанков.

**Обязательные опции:**
- Чанки с `library()`: `#| message: false` — кроме `107-beyond_base_r.qmd` и `110-tidyverse_basic.qmd` (намеренно).
- Чанки с импортом (`read_csv`, `vroom`, `read_excel`): `#| warning: false` — кроме первого `vroom::vroom()` в `030-import_data.qmd` (намеренно).
- Чанки с намеренной ошибкой: `#| error: true`.

**Подписи к рисункам** — через `#| fig-cap:`, не через старый `fig.cap=` в заголовке чанка. Метка рисунка начинается с `fig-`:
```r
#| label: fig-histogram
#| fig-cap: "Гистограмма роста супергероев"
```

### 19. Callout-блоки

| Тип | Заголовок | Когда использовать |
|---|---|---|
| `callout-important` | `## *Осторожно:* описание` | Типичные ошибки, подводные камни, неочевидное поведение R |
| `callout-warning` | `## *Для продвинутых:* описание` | Углублённые детали, которые начинающий может пропустить |
| `callout-tip` | `## *Полезное:* описание` | Дополнительные сведения, связи с другими темами, лайфхаки |
| `callout-note` | `## *Практика N:* описание` | Упражнения для самостоятельного выполнения (см. ниже) |
| `callout-note` | `## В разработке` | Заглушки для незаписанных разделов |
| `callout-caution` | `## Время мемов` | Мемы и юмористические отступления |

Слово-тип (*Осторожно*, *Для продвинутых*, *Полезное*, *Практика*) — курсивом, за ним двоеточие.

**Частые ошибки:**
- `callout-important` вместо `callout-warning` для «Для продвинутых» и наоборот — разные смыслы: *Осторожно* = опасность/ошибка, *Для продвинутых* = опциональная глубина.
- `callout-note` вместо `callout-tip` для «Полезное».
- Заголовок без курсива: `## Осторожно:` вместо `## *Осторожно:*`.

#### Система упражнений

Упражнения размещаются inline после ключевых подразделов. Нумерация — порядковая внутри главы (1, 2, 3...). Каждое упражнение ссылается на решение в `920-solutions.qmd`.

**Формат упражнения:**

```markdown
::: {.callout-note #ex-chaptername-N}
## *Практика N:* краткое описание

Текст задания.

[Решение →](#sec-sol-chaptername-N)
:::
```

**Якоря:** `#ex-vector-1`, `#ex-vector-2` — в файле с упражнением; `#sec-sol-vector-1` — в `920-solutions.qmd`.

**Формат решения в `920-solutions.qmd`:**

```markdown

### 20. Blockquotes (`>`)

Используются для лёгких блоков без заголовка:
- Короткие определения
- Лёгкие советы, не требующие callout'а
- Подсказки к заданиям
- Мнемоники и неформальные выводы
- Стихи, мемы, цитаты

**Не использовать** для предупреждений об ошибках, полезных дополнений или упражнений — для этого callout'ы.

---

### 21. Сноски

Для побочных замечаний, которые прерывали бы основной текст.

**Формат имени:** `[^NNN-filename-N]` — например: `[^010-intro-3]`.

**Сноска vs callout vs blockquote:**
- Можно пропустить без потери смысла → сноска
- Читатель не должен пропускать → `callout-tip`
- Короткое определение или мнемоника → blockquote

---

### 22. Cross-references

Якоря: `{#sec-label}`  
Ссылки: `@sec-label` (не `\@ref()`)

Стандартные формулировки:
```
(см. @sec-label)
подробнее разбираем в @sec-label
```

Не писать «на рисунке выше» или «в разделе ниже» — использовать `@sec-label`, `@fig-label`.

### 23. Изображения

**Именование:** `NNN-chaptername_описание.ext`  
**Расширения** — строчными: `.png`, `.jpg`, `.gif`

**Alt-текст — обязателен.** Описывать что изображено и что это означает для читателя:
```
❌ ![](images/plot.png)
❌ ![График](images/plot.png)
✅ ![Гистограмма роста супергероев: правосторонняя асимметрия, большинство значений 170–200 см](images/plot.png)
```

⚠️ **Текущее состояние:** большинство изображений в книге без alt-текста — высокий приоритет при редактировании.

### 24. Ссылки на внешние ресурсы

Текст ссылки описывает содержимое, не направление:
```
❌ подробнее можно почитать [здесь](https://...)
✅ подробнее в [Advanced R](https://...)
✅ скачайте [файл с данными](https://...)
```

### 25. Цитирование

Книга адресована академической аудитории --- используется стиль **APA 7-е издание**.

**Конфигурация** (`_quarto.yml`):
```yaml
bibliography: references.bib
csl: apa.csl
```
Файл `apa.csl` лежит в корне проекта. Не менять на другой стиль без веской причины.

**Форматы ссылок в тексте:**

| Ситуация | Синтаксис Quarto | Результат (APA) |
|---|---|---|
| Скобочная ссылка | `[@key]` | (Wickham, 2010) |
| Нарративная ссылка | `@key` | Wickham (2010) |
| Несколько источников | `[@key1; @key2]` | (Adler, 2010; Wilkinson, 2005) |
| Без имени автора | `[-@key]` | (2010) |

**Когда какой формат:**
- В конце предложения/абзаца --- скобочный: «...изложенный в одноимённой книге [@gg]»
- Когда автор --- субъект предложения --- нарративный: «@wickham16 описывают подход...»

**Оформление текста вокруг ссылки:**
- Ссылка идёт перед точкой, не после: «...результатах [@eduiq].», не «...результатах. [@eduiq]»
- Полное название цитируемой работы при первом упоминании --- курсивом в кавычках: «в статье *"A Layered Grammar of Graphics"* [@layered-grammar]»

**Файл `references.bib`:** ключи должны быть в формате `авторгод` (строчными, без пробелов): `wickham16`, `nuzzo14`. Исключения для устойчивых ключей: `gg`, `layered-grammar`, `normality` --- не переименовывать, они уже используются в тексте.

**Гиперссылка vs цитата:**

| Тип источника | Оформление |
|---|---|
| Журнальная статья, книга (есть DOI или ISBN) | `[@key]` — добавить в bib |
| Документация пакета/инструмента | инлайн-гиперссылка |
| Блог, туториал, Stack Overflow | инлайн-гиперссылка |
| Интерактивное демо (Shiny, D3) | инлайн-гиперссылка |
| Wikipedia | инлайн-гиперссылка (не академический источник) |
| Данные для скачивания | инлайн-гиперссылка |

Критерий: если источник можно найти по DOI или он прошёл рецензирование --- в bib и `[@key]`. Во всех остальных случаях --- гиперссылка с содержательным текстом (§24).

**Неиспользуемые записи в bib** --- не удалять записи, релевантные темам книги (p-value, нормальность, LM, χ²) --- они подготовлены для ещё непроцитированных мест в главах 305--370.

**Проверка перед публикацией:**
```bash
# Найти все cite-ключи, использованные в главах
grep -roh '@[a-z][a-z0-9_-]*' *.qmd | grep -v '@sec-\|@fig-\|@tbl-\|@eq-' | sort -u
```

### 26. Условный контент HTML vs PDF

Для контента, который не работает в PDF (анимации, интерактивность):
```r
if (knitr:::is_latex_output()) {
  cat("\n\nК сожалению, в PDF нельзя вставить ..., смотрите онлайн-версию книги)")
} else {
  # интерактивный/gif контент
}
```
Текст в `cat()` — стандартная формулировка, не изменять.

### 27. Незавершённые места

Заметки автора — HTML-комментарии:
```
<!--# Сюда картинку с квартилями! -->
```
Не попадают в рендер. Перед публикацией — найти и закрыть все:
```bash
grep -n "<!--#" NNN-chapter.qmd
```
